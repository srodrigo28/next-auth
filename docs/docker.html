Módulo Inicial

Visão geral do Docker

Docker é uma plataforma de containerização que permite empacotar, distribuir e executar aplicações em ambientes isolados chamados containers. Em vez de depender da configuração da máquina, a aplicação leva junto tudo o que precisa para funcionar.

Finalidade

Serve para transformar uma aplicação em uma imagem padronizada, com código, bibliotecas, dependências e configurações, facilitando a criação do mesmo container em desenvolvimento, homologação e produção.

Princípio operacional

Criar ambientes isolados e previsíveis. Cada sistema roda no seu próprio container, sem interferir nos outros, mantendo consistência e reduzindo erros de configuração.

Definição objetiva

Docker permite empacotar a aplicação em uma imagem reproduzível. A partir dessa imagem, o mesmo container pode ser iniciado em ambientes diferentes, com menos variação de configuração.

PASSO 1

O que o Docker resolve?

Esta seção apresenta os problemas operacionais que o Docker ajuda a reduzir em desenvolvimento, teste e publicação.

Diferença entre ambientes

Evita o famoso “na minha máquina funciona”. O mesmo container roda igual em qualquer máquina, reduzindo falhas entre dev, teste e produção.

Conflito de dependências

Permite rodar aplicações com versões diferentes de Node, PHP, Java, Python ou banco sem que uma quebre a outra.

Deploy mais rápido

Com a imagem pronta, a aplicação sobe mais rápido e com menos improviso. Isso reduz retrabalho, erros manuais e acelera a entrega.

Padronização operacional

Facilita treinamento, manutenção e escalabilidade, porque todos seguem o mesmo modelo de empacotamento e execução.

Em uma frase para apresentação

Docker resolve inconsistência de ambiente, conflito de dependências e lentidão no deploy, criando uma forma padronizada, rápida e previsível de rodar aplicações.

PASSO 2

Requisitos antes de instalar

Antes da instalação, valide se a máquina tem folga suficiente para executar containers sem travar o sistema. Essa checagem evita uma configuração que funciona, mas entrega uma experiência ruim.

Processador

Recomendado: CPU 64 bits com virtualização habilitada. Em desktop, pense em Intel Core i5, AMD Ryzen 5 ou superior. No Mac, Apple Silicon M1 ou mais novo entrega ótima experiência.

Memória RAM

Mínimo prático: 8 GB. Recomendado para aula, API, banco e navegador ao mesmo tempo: 16 GB. Abaixo disso, o Docker até abre, mas a multitarefa fica limitada.

Espaço em disco

Reserve pelo menos 20 GB livres. Recomendado: 40 GB ou mais, porque imagens, volumes, cache de build e logs crescem rápido durante testes e estudos.

Windows

Use Windows 10 ou 11 em 64 bits, com WSL 2 ativado. Também vale confirmar na BIOS se a virtualização (VT-x ou AMD-V) está habilitada.

Linux

Prefira uma distro 64 bits atualizada, como Ubuntu 22.04+, Debian 12+, Fedora ou equivalente. É importante ter acesso a sudo para instalar pacotes e iniciar o serviço.

macOS

Use macOS recente em chip Intel ou Apple Silicon. Antes de baixar, confirme qual arquitetura do Mac está em uso para escolher o instalador correto.

Escolha apenas uma trilha de instalação

Windows, Linux e macOS são caminhos alternativos. A partir daqui, escolha somente a trilha correspondente ao sistema operacional da máquina.

Regra prática para laboratório

Se a máquina tiver CPU 64 bits, virtualização ativa, 16 GB de RAM e pelo menos 40 GB livres, já é um ambiente confortável para aprender Docker, testar imagens e subir pequenos stacks locais.

PASSO 3

Instalação no Windows

Escolha esta trilha se a máquina usar Windows. O caminho mais estável é usar o Docker Desktop com backend em WSL 2, que entrega interface gráfica, engine, Compose e integração com terminal Linux.

1

Ative o WSL 2

Abra o PowerShell como administrador e habilite o subsistema Linux. Reinicie se o Windows solicitar. 

wsl --install
2

Instale o Docker Desktop

Baixe o instalador para Windows, execute o setup e mantenha marcada a opção de usar WSL 2. Ao fim, abra o Docker Desktop para concluir a configuração inicial.

3

Confirme a integração

Nas configurações do Docker Desktop, valide se a integração com sua distro WSL está ativa. Isso permite usar docker direto do terminal Linux.

4

Teste a instalação

Rode um container de teste para validar engine, download de imagem e execução local. 

docker version
docker run hello-world

Ponto técnico importante

No Windows, o Docker não roda sozinho como no Linux puro: ele usa WSL 2 para entregar um ambiente Linux por baixo. Essa camada é o motivo de a virtualização precisar estar ativa.

PASSO 4

Instalação no Linux

Escolha esta trilha se a máquina usar Linux. O fluxo mais comum é instalar o Docker Engine diretamente no sistema. O exemplo abaixo funciona bem como referência para distribuições baseadas em Ubuntu e Debian.

1

Atualize o sistema

Antes de instalar, atualize a lista de pacotes e garanta dependências básicas de certificados e download.

2

Instale o engine

Após adicionar o repositório oficial do Docker, instale o engine, o CLI, o Buildx e o plugin do Compose.

3

Ative e valide

Suba o serviço no boot, rode um container de teste e, se quiser usar sem sudo, adicione seu usuário ao grupo docker.

Sequência prática (Ubuntu / Debian)

Terminal 
sudo apt update
sudo apt install ca-certificates curl
# adicione o repositório oficial do Docker
sudo apt update
sudo apt install docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
sudo systemctl enable --now docker
sudo docker run hello-world

Observação importante

No Linux, o Docker costuma consumir menos overhead porque roda nativamente no kernel. Por isso ele é a base mais comum para servidores e ambientes de produção.

PASSO 5

Instalação no macOS

Escolha esta trilha se a máquina usar macOS. A instalação padrão também é feita com Docker Desktop. O ponto principal é baixar o instalador correto para o chip Intel ou Apple Silicon.

1

Identifique o chip

Em Sobre Este Mac, confira se a máquina usa Intel ou Apple Silicon. Isso define qual pacote .dmg você deve baixar.

2

Instale o aplicativo

Abra o arquivo baixado, arraste o Docker para a pasta Applications e inicie o app pela primeira vez para concluir permissões e componentes auxiliares.

3

Aceite permissões

O macOS pode pedir senha do sistema para instalar componentes de rede e integração com o terminal. Em Macs Intel, o sistema também pode sugerir Rosetta 2.

4

Valide no terminal

Depois do aplicativo iniciar, abra o Terminal e confirme versão e execução do container de teste. 

docker version
docker run hello-world

Observação técnica

No macOS, o Docker Desktop também cria uma camada Linux controlada pelo aplicativo. Para o usuário, a experiência parece nativa, mas por baixo existe uma VM otimizada para rodar os containers.

PASSO 6

Dockerfile, imagem e build

Aqui começa o fluxo técnico de empacotamento: primeiro você descreve como a imagem deve ser construída, depois executa o build e gera um artefato imutável para rodar em qualquer ambiente compatível.

Dockerfile

É um arquivo texto com instruções de build como FROM, RUN, COPY, ENV e CMD. Ele define como a imagem será montada a partir de uma base, quais dependências serão instaladas e como a aplicação será inicializada.

Imagem

É o artefato gerado pelo processo de build. A imagem é versionável, imutável e reutilizável, contendo sistema base, runtime, bibliotecas, dependências e a configuração necessária para iniciar o processo principal da aplicação.

Encadeamento correto

Em termos técnicos, a sequência é: Dockerfile define o build, docker build gera a imagem, e a imagem depois pode ser instanciada como um ou mais containers. O Dockerfile não é o container em si e não substitui a definição de múltiplos services em compose.yaml.

Leitura correta para documentação técnica

Para documentação, trate o Dockerfile como definição de construção de imagem e de processo de build. Isso evita ambiguidade e deixa claro que ele é um artefato técnico versionado no repositório.

Exemplos de Dockerfile

Dockerfile

Python + Flask

FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

ENV FLASK_PORT=5000

EXPOSE 5000

CMD ["python", "main.py"]
Dockerfile

Java + Postgres

FROM eclipse-temurin:21-jre-alpine

WORKDIR /app

COPY target/app.jar app.jar

ENV SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/appdb

EXPOSE 8080

CMD ["java", "-jar", "app.jar"]
Dockerfile

Node.js + MySQL

FROM node:20-alpine

WORKDIR /app

COPY package*.json ./
RUN npm ci --omit=dev

COPY . .

ENV DB_HOST=mysql
ENV DB_PORT=3306

EXPOSE 3000

CMD ["npm", "start"]

PASSO 7

Container em execução (runtime)

Depois do build, a imagem passa a ser executada. O container é a instância em runtime desse artefato, com processo, rede e camada de escrita próprios.

Definição técnica

Um container é um processo isolado criado a partir de uma imagem. Ele compartilha o kernel do host, mas roda com isolamento de filesystem, rede e recursos. Por isso, não deve ser descrito como máquina virtual completa, e sim como unidade de execução empacotada.

Diferença crítica: imagem x container

A imagem é estática e reutilizável. O container é dinâmico e representa a execução dessa imagem em um determinado momento. Uma única imagem pode originar vários containers, cada um com seu próprio estado de runtime.

Comando para criar e iniciar

Terminal 
# Cria o container do Flask e publica a porta HTTP local
docker run -d -p 8000:5000 --name painel-flask-app painel-flask:latest

Comandos úteis de inspeção

Terminal 
# Lista containers em execução
docker ps

# Consulta logs do container
docker logs painel-flask-app

Processo principal

Cada container nasce para executar um processo principal e seu ciclo de vida depende desse processo.

Camada gravável

O container ganha uma camada de escrita temporária sobre a imagem, útil para runtime mas descartável sem volume.

Rede e portas

A comunicação externa normalmente depende de mapeamento de portas, como 8000:5000 em um serviço Flask.

Efêmero por padrão

Se o container for removido, o estado local se perde, a menos que os dados estejam em volumes ou storage externo.

PASSO 8

Docker Compose, arquivo compose.yaml, volumes e redes

Depois do runtime básico, a próxima camada é a definição e a execução de múltiplos serviços. Aqui entra a diferença entre o arquivo declarativo da stack e a ferramenta que interpreta esse arquivo.

Arquivo de composição

O arquivo de definição normalmente aparece como compose.yaml, compose.yml ou, em projetos legados, docker-compose.yml. Ele declara services, image, build, variáveis, portas, volumes e networks em YAML.

Docker Compose (CLI)

É a ferramenta que interpreta o arquivo e reconcilia o ambiente. No padrão atual, o comando é docker compose. Em materiais mais antigos, você ainda pode encontrar docker-compose.

Volume

Define persistência de dados fora do ciclo de vida do container. Isso é essencial para banco de dados, uploads, arquivos de cache e qualquer informação que não pode ser perdida ao recriar services.

Rede (network)

Define a conectividade entre services. Com uma network interna, o Flask pode falar com o MySQL usando nomes de service sem expor tudo externamente.

Separação conceitual importante

Em um documento técnico, vale manter a distinção explícita: o arquivo YAML descreve o estado desejado de uma aplicação com múltiplos serviços; o Docker Compose é a ferramenta que lê essa definição e aplica as mudanças necessárias no ambiente.

Tabela comparativa dos conceitos

Visão rápida do papel de cada elemento, do build até a execução da stack.

Elemento Tipo Função técnica Quando entra Arquivo ou comando associado
Dockerfile Arquivo de build Descreve como a imagem será construída. Build Dockerfile e docker build
Imagem Artefato imutável Empacota runtime, código e configuração. Pós-build docker build, docker pull, docker image ls
Container Instância em execução Executa a imagem como processo isolado. Execução docker run, docker ps
compose.yaml Arquivo declarativo Declara a stack com services, volumes e networks. Definição da stack compose.yaml, compose.yml ou docker-compose.yml
Docker Compose Ferramenta CLI Aplica o arquivo e sobe ou atualiza a stack. Execução da stack docker compose up, docker compose down

Tabela complementar da stack

Aqui ficam os blocos operacionais mais usados dentro do arquivo de composição.

Conceito O que representa Exemplo em compose.yaml Impacto prático
service Um processo ou componente da aplicação. flask_app, mysql_db Separa responsabilidades e facilita escalar ou reiniciar partes isoladas.

Exemplo de stack Flask + MySQL

compose.yaml 
services:
  flask_app:
    build:
      context: .
    command: python main.py
    ports:
      - "8000:5000"
    env_file:
      - .env
    environment:
      FLASK_PORT: 5000
      SECRET_KEY: troque-por-uma-chave-segura-aleatoria
      DB_HOST: mysql_db
      DB_PORT: 3306
      DB_USER: eduportal_user
      DB_PASSWORD: senha_app
      DB_NAME: eduportal
    depends_on:
      - mysql_db
    networks:
      - app_net

  mysql_db:
    image: mysql:8
    environment:
      MYSQL_DATABASE: eduportal
      MYSQL_USER: eduportal_user
      MYSQL_PASSWORD: senha_app
      MYSQL_ROOT_PASSWORD: senha_root
    volumes:
      - dados_mysql:/var/lib/mysql
    networks:
      - app_net

volumes:
  dados_mysql:

networks:
  app_net:

PASSO 8A

Docker Compose x Dockerfile

Mesmo que muita gente fale "Docker Composer", o nome correto e Docker Compose. A duvida mais comum aqui e separar o arquivo de build do arquivo que organiza a stack.

Resposta curta

Dockerfile

Define como a imagem sera montada: base, dependencias, copia de arquivos, variaveis e comando inicial.

compose.yaml

Define como varios servicos vao rodar juntos: app, banco, portas, volumes, redes e dependencias.

docker compose

E o comando que le o YAML, cria os recursos e sobe ou derruba a stack declarada.

Quando usar Dockerfile

Quando voce precisa montar a imagem da aplicacao.

Quando quer instalar pacotes, copiar codigo ou definir o comando de start.

Quando o problema esta na construcao do ambiente da aplicacao.

Quando usar Compose

Quando voce precisa subir app, banco, cache ou fila em conjunto.

Quando quer mapear portas, conectar redes e declarar volumes em um unico arquivo.

Quando o problema esta na orquestracao local ou na stack de desenvolvimento.

Comparacao direta

Se a sua pergunta for "qual arquivo eu altero?", use esta regra simples.

Cenario 1

Quero mudar a versao do Python, instalar pacotes ou trocar o comando inicial.

Edite o Dockerfile.

Cenario 2

Quero subir junto banco, porta, volume, variaveis e dependencia entre servicos.

Edite o compose.yaml.

Cenario 3

Quero automatizar build e subida da stack com um comando so.

Use build: . no compose e rode docker compose up --build.

Regra que evita confusao

Compose nao substitui o Dockerfile. Na pratica, o Compose pode apontar para build: ., mas isso apenas diz para ele usar o Dockerfile do projeto antes de subir os servicos. Um monta a imagem; o outro organiza a stack.
